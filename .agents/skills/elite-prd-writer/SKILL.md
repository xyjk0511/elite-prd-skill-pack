# Elite PRD Writer

## 目标

把用户的想法、草稿或现有需求转成工程可开工的产品需求文档，而不是写成长篇空话。

完成标准：

- 设计可以据此出原型。
- 前端可以识别页面、组件、交互状态、字段校验。
- 后端可以识别对象、接口、权限、状态流转。
- QA 可以据此生成测试用例。
- 数据可以据此完成埋点与指标口径。
- 团队可以明确知道本期做什么、明确不做什么。

## 默认语言

默认使用中文输出。文件名、事件名、指标键名、API 路径、字段名、代码、命令和目录名可以保留英文。

## 工作边界

使用本技能处理：

- 从 0 创建 PRD / 产品需求书 / 产品规格。
- 把模糊想法、issue、会议纪要、聊天记录补全为需求文档。
- 重写不够专业或不够完整的 PRD。
- 生成 MVP 范围、业务规则、验收标准、埋点方案。
- 为设计、研发、测试和数据建立统一需求基线。

不要使用本技能处理：

- 用户明确要求直接写代码。
- 用户只要技术设计、测试用例、开发任务、tickets 或 API stub。
- 用户只是问一个小 bug 或微调项，且不需要文档。

遇到边界外任务，改用 `requirements-clarity`、`prd-auditor`、`implementation-handoff`、`qa-generator` 或专门技术设计流程。

## 工作流程

1. 先检查仓库已有上下文：`AGENTS.md`、`README.md`、`docs/`、`tasks/`、历史 PRD、issue、roadmap、设计或架构文档。
2. 如果仓库已有命名和目录规范，优先遵循仓库规范。
3. 判断需求清晰度。只有缺失信息会实质影响 PRD 内容时才提问。
4. 用户要求“直接写”时不要阻塞；用显式假设继续。
5. 默认输出完整 PRD；用户要求轻量版时输出简版 PRD。
6. 输出 `PRD Packet`，供 `prd-auditor`、`implementation-handoff`、`qa-generator` 使用。
7. 如果用户要求创建文件，或仓库已有文档目录规范，保存到文件。

## 澄清策略

最多提 7 个问题，按影响优先级排序：

1. 目标用户是谁。
2. 解决什么核心问题。
3. 业务 / 产品目标是什么。
4. 核心用户流程是什么。
5. 本期必须做什么。
6. 本期明确不做什么。
7. 成功指标是什么。
8. 有哪些技术、合规、时间、数据、运营约束。

如果需要先做系统澄清，改用 `requirements-clarity`。

信息不足但可以推进时，先写：

```text
以下 PRD 基于这些假设：
1. 假设目标用户为……
2. 假设本期目标为……
3. 假设暂不包含……
4. 假设成功指标以……为准
```

不要把假设写成已确认事实。

## 与其他技能的衔接

- 输入来自 `requirements-clarity` 时，优先使用其中的 `Requirements Packet`，并把未确认项写入 PRD 的“待确认问题”。
- PRD 生成后，默认建议进入 `prd-auditor` 做开工审计。
- 只有 `prd-auditor` 给出“可以开工”或“补充后开工且无 P0 阻塞”时，才进入 `implementation-handoff`。
- `qa-generator` 应消费已通过审计的 PRD；如果 PRD 缺少验收标准、状态机或权限规则，先回到本技能补 PRD。

## 输出结构

默认使用 `references/prd-template.md`。至少包含：

1. 文档信息
2. 一句话概述
3. 背景与问题
4. 目标用户与使用场景
5. 产品目标
6. 成功指标
7. 本期范围
8. 非本期范围
9. 用户流程
10. 页面 / 模块结构
11. 功能需求详述
12. 业务规则
13. 状态流转
14. 权限设计
15. 字段与数据要求
16. 埋点与指标口径
17. 非功能需求
18. 异常与边界情况
19. 验收标准
20. 风险与依赖
21. 待确认问题

行为改变型功能必须包含埋点和指标。需要事件命名、漏斗或字段模板时读取 `references/metrics-and-events.md`。

## 功能需求写法

每个重要功能使用：

```md
### FR-xxx 功能名称

- 目标用户：
- 优先级：P0 / P1 / P2 / P3
- 需求描述：
- 前置条件：
- 触发条件：
- 主流程：
- 分支流程：
- 异常流程：
- 业务规则：
- 字段规则：
- 权限规则：
- 验收标准：
- 埋点需求：
```

所有 P0 功能必须有可测试验收标准。优先使用 Given-When-Then。

## 结果落盘

默认推荐路径：`docs/prd/prd-[feature-name].md`。

文件名使用 kebab-case。如果仓库已有自己的目录或命名方式，遵循仓库规则，不强行覆盖。

需要确定性写文件时，使用 `scripts/save_doc.py`：

```bash
python .agents/skills/elite-prd-writer/scripts/save_doc.py --path docs/prd/prd-[feature-name].md < draft.md
```

## PRD Packet

输出或落盘 PRD 后，附上交接块：

```md
## PRD Packet

- PRD 文件路径：
- 功能名称：
- 版本 / 状态：
- 本期范围：
- 非本期范围：
- P0 功能列表：
- 核心状态机：
- 权限边界：
- 关键字段：
- 埋点事件：
- 待确认问题：
```

## 严格禁止

- 不要在 PRD 生成过程中直接开始实现代码。
- 不要跳过范围边界、异常流程、权限设计、埋点与指标。
- 不要把假设伪装成事实。
- 不要输出只有 happy path 的 PRD。
- 不要把技术方案写成唯一实现方案，除非用户明确要求。

## 结束前自检

输出前读取 `references/quality-checklist.md` 或按其中标准静默自检。若设计、前端、后端、QA、数据任一角色无法开工，先补全文档再输出。
