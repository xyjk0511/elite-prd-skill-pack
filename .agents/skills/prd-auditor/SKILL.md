# PRD Auditor

## 目标

判断一份 PRD 是否能让设计、研发、测试、数据开工，并给出最小可执行修复方案。

完成标准：

- 先给结论：可以开工 / 补充后开工 / 暂不建议开工。
- 用评分说明缺口，不只给主观评价。
- 找出阻塞实现、测试、设计或数据埋点的缺失项。
- 给出可直接替换的章节、表格或验收标准片段。

## 使用场景

使用本技能处理：

- “审查这份 PRD”
- “这份需求能不能给研发开工”
- “哪里不完整”
- “帮我查漏补缺”
- “给 PRD 打分并修一下”

不要处理：

- 从零创建 PRD：改用 `elite-prd-writer`。
- 需求还非常模糊：改用 `requirements-clarity`。
- 已批准 PRD 的任务拆解：改用 `implementation-handoff`。
- 基于 PRD 生成测试用例：改用 `qa-generator`。

## 工作流程

1. 读取用户提供的 PRD 或仓库文件。
2. 检查仓库 `AGENTS.md` 和已有文档规范。
3. 按 `references/audit-rubric.md` 逐项评分。
4. 区分阻塞问题、重要缺口和可延后优化。
5. 对缺失章节给出可复制修复片段。
6. 输出 `Audit Packet`，明确是否达到开工标准，以及是否允许进入 `implementation-handoff` 和 `qa-generator`。

## 与其他技能的衔接

- 结论为“可以开工”：允许进入 `implementation-handoff` 和 `qa-generator`。
- 结论为“补充后开工”：如果无 P0 阻塞，可进入 `implementation-handoff`，但 handoff 必须保留待确认问题；如果有 P0 阻塞，先回到 `elite-prd-writer` 修 PRD。
- 结论为“暂不建议开工”：停止后续 handoff 和 QA，回到 `requirements-clarity` 或 `elite-prd-writer`。
- 用户要求“直接修”时，输出可替换片段，并说明修复后需要重新审计。

## 审计原则

- 发现问题必须说明影响哪个角色：产品、设计、前端、后端、QA、数据、运营。
- 不使用“再详细一点”这类空建议；必须给出具体补法。
- 对 P0 功能，必须检查验收标准、异常流程、字段规则、权限和埋点。
- 若用户要求“直接修”，输出改写后的章节，不只输出评论。

## 严格禁止

- 不要把审计变成从零重写，除非用户要求。
- 不要只做文字润色。
- 不要忽略状态流转、权限、字段和验收标准。
- 不要在未检查 PRD 的情况下泛泛打分。
